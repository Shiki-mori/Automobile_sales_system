```text

Customer(

Service_Item(
```

🔴 业务规则 A

创建订单时，车辆必须是“在库”，并变为“已锁定”

实现方式：

1️⃣ 设计层：
Car.status 字段记录状态

2️⃣ 约束：
Sales_Order.vin UNIQUE
👉 保证一辆车只能被一个订单使用

3️⃣ 执行层（后续实现）：
使用触发器 trg_lock_car_on_order
INSERT订单 → UPDATE Car.status = '已锁定'

🔴 业务规则 B

订单完成 → 车辆变为“已售出”

实现方式：

1️⃣ 依赖字段：
Sales_Order.status
Sales_Order.delivery_time

2️⃣ 触发器：
UPDATE Sales_Order.status = '已完成'
→ UPDATE Car.status = '已售出'

🔴 业务规则 C

库存必须实时准确

实现方式：

1️⃣ 不存库存数量字段 ❗（关键设计点）
👉 不建立Model.stock_count

2️⃣ 改为动态统计：
SELECT model_id, COUNT(*)
FROM Car
WHERE status = '在库'
GROUP BY model_id;

👉 保证：
无冗余
无数据不一致问题

## 设计亮点

- 产品与实体分离  
车型（Model） vs 车辆（Car）
- 订单绑定具体车辆（VIN）  
符合真实业务
- 使用唯一约束实现“一车一单”  
Sales_Order.vin UNIQUE
- 库存动态计算  
避免冗余字段  
- 复杂信息拆表  
订单明细 / 服务明细

## 3NF规范化分析

根据 `01_create_schema.sql` 的表结构，所有关系模式均符合**第三范式（3NF）**，具体分析如下：

### 1NF（第一范式）

所有表的每个字段都是**原子值**，不可再分：

- `Customer.address`：完整地址字符串，非省/市/区分列（业务上视为单一属性）
- `Model.guide_price`：使用 `DECIMAL` 精确存储金额
- `Car.status`：使用枚举值，避免多值存储

### 2NF（第二范式）

所有非主属性**完全函数依赖**于主键，无部分依赖：

- `Model` 表：所有属性依赖 `model_id`，不存在仅依赖 `brand_id` 的属性（`brand_id` 只是外键）
- `Car` 表：`color`、`engine_no` 等属性均依赖 `vin`（主键），而非部分依赖 `model_id`
- `Sales_Order` 表：`total_amount`、`status` 等完全依赖 `order_id`，而非部分依赖 `customer_id` 或 `vin`

### 3NF（第三范式）

所有非主属性**直接函数依赖**于主键，无传递依赖：

| 表名 | 分析 |
| ---- | ---- |
| **Brand** | `brand_id → name`，无传递依赖 ✓ |
| **Model** | `model_id → series_name/year/config_name`，`guide_price` 直接依赖本表主键，不存在 `model_id → brand_id → brand_name` 的传递依赖（品牌名在Brand表维护）✓ |
| **Car** | `vin → color/engine_no/status`，`purchase_price` 直接依赖本车VIN，非依赖车型计算得出 ✓ |
| **Customer** | `customer_id → name/phone/address`，无传递依赖 ✓ |
| **Employee** | `employee_id → name/job_number/role`，`manager_id` 是自引用外键，不构成传递依赖 ✓ |
| **Sales_Order** | `order_id → total_amount/status/create_time`，`total_amount` 由订单明细汇总计算，不作为存储字段避免冗余，实际表中记录为快照值，且直接依赖主键 ✓ |
| **Order_Item** | `item_id → type/amount`，直接依赖 ✓ |
| **Service_Order** | `service_id → service_type/total_cost/status`，直接依赖 ✓ |
| **Service_Item** | `item_id → name/quantity/price/amount`，直接依赖 ✓ |
| **Intention** | `intention_id → level/note/next_contact_time`，直接依赖 ✓ |

### 关键3NF设计决策

#### 决策1：库存数量不存储在Model表中

```sql
-- 反范式做法（不符合3NF）
ALTER TABLE model ADD stock_count INT;  -- ❌ 依赖统计结果，非直接依赖主键

-- 本设计做法（符合3NF）
-- 通过视图动态统计，见 03_views.sql:v_inventory_summary
```

#### 决策2：订单总金额存储与明细分离

```sql
-- Sales_Order.total_amount 存储快照值
-- Order_Item.amount 存储各明细金额
-- 二者通过触发器/存储过程保持一致
-- 属于合理冗余（controlled redundancy），保证查询性能的同时通过应用层/触发器维护一致性
```

#### 决策3：客户分级不存储字段

```sql
-- 不存储 Customer.level 字段
-- 通过视图 v_customer_value 动态计算：
-- CASE WHEN total_amount < 100000 THEN '普通客户' ... END
```

### 结论

所有10个关系模式均满足 **3NF**，无传递依赖，无部分依赖，属性原子化。通过外键约束实现参照完整性，通过触发器实现业务规则，通过视图实现统计查询，整体设计兼顾规范性与实用性。
